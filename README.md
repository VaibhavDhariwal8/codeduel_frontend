## About

CodeDuel is a real-time, head-to-head competitive coding platform — chess.com meets LeetCode. Two players are matched on skill, dropped into the same problem under a shared countdown, and can watch each other's hidden-test pass count climb in real time. Neither player can ever see the other's code mid-match; the opponent's solution only unlocks for review once the duel ends. Behind that single mechanic sits a full platform: skill-based matchmaking with a cold-start fallback, a self-hosted sandboxed execution engine running untrusted code across four languages, an Elo-based ranking system, a friends/social layer, exact-hash plagiarism detection feeding a real admin moderation queue, and a Gemini-powered AI coach that reviews your specific approach after every match.

## Features

<table>
<tr>
<td width="50%" valign="top">

### Core Duel Loop

- Quick Match with skill-based pairing and a rating-band-widening fallback so early/low-traffic periods don't produce indefinite queue times
- Custom Duel via a shareable link — first person to open and accept it gets pulled into the match live, no refresh required
- Synced countdown timer, Monaco-powered editor with language tabs (Python, JavaScript, Java, C++), resizable split panels, local draft auto-save
- Run against visible sample tests at any time; Submit grades against the full hidden suite
- Live opponent progress broadcast — pass/fail counts only, their code is never transmitted
- First-to-solve, time-expiry tiebreak by test count, draws, and forfeits all resolve correctly server-side

### Authentication

- Email/password and GitHub OAuth via Supabase Auth
- `public.users` profile rows are FK-linked directly to `auth.users` — no separate UID-mirroring step
- Every REST call and Socket.IO connection is JWT-verified server-side before touching any duel or matchmaking logic
- Banned accounts are rejected at both the REST middleware and the socket handshake, and a live ban force-disconnects any already-open session

</td>
<td width="50%" valign="top">

### AI Coach

- Post-match "autopsy" reviewing the _requesting player's own code_ against their opponent's, generated by the Gemini API
- Strict JSON response schema: outcome summary, identified bottleneck, conceptual gap, a Socratic hint (hidden until clicked, never a corrected solution), and a step-by-step breakdown
- System-instruction-locked persona — the model is explicitly forbidden from ever returning corrected code, only guiding questions
- Cached per submission so repeat visits to a Result screen never re-bill the same review

### Anti-Cheat & Trust

- Paste-blocking inside the editor during ranked duels, with an honest on-screen notice rather than a silent failure
- Submission rate-limiting per user
- Exact-hash plagiarism detection — every submission's code is hashed and checked against other accounts' submissions on the same problem, with hits written to a dedicated flagged-pairs table
- A report queue that gives admins real signal beyond the reporter's free-text claim: total reports against the user, how many _distinct_ people filed them, flagged code-similarity hits, and the reporter/reported head-to-head record

</td>
</tr>
<tr>
<td width="50%" valign="top">

### Ranking & Social

- Elo-style rating engine, scaled by opponent strength, with full rating-change history
- Bronze through Grandmaster tiers, rendered as a shield-shaped badge on profile pages
- Friend requests, accept/decline, and a friends-only leaderboard
- Live rematch requests negotiated over the same socket connection both players are already using — no new invite link needed
- Match history with a rating-over-time chart, per-difficulty win-rate breakdown, recent-form strip, longest win streak, and most-played opponent

</td>
<td width="50%" valign="top">

### Admin Tools

- Report review queue gated server-side by an `is_admin` flag, independent of any client-side UI
- Dismiss or Ban actions per report
- Banning a user: force-disconnects every live socket they hold via a personal Socket.IO room, auto-resolves any in-progress match as a forfeit in the opponent's favor, and immediately blocks further REST/socket access

</td>
</tr>

### UI/UX

- Dark, purpose-built design system (Tailwind v4, CSS-first theme tokens) — not default Tailwind styling
- Match-found ceremony with avatar slide-in, a VS flash, and a 3-2-1 countdown before entering the arena; full `prefers-reduced-motion` support throughout
- Pixel-art avatars (DiceBear), deterministically seeded per username
- A themed scrollbar, sticky stat sidebars on data-heavy pages, and a lightweight floating walkthrough widget answering common "how does this work" questions without needing a live chatbot
- Responsive layout, dark-mode-only by design to match the competitive-judge aesthetic

## Architecture

```
┌─────────────────────────────────────────────┐
│              FRONTEND (Vercel)              │
│      React + Vite + Tailwind + Monaco       │
└──────────────────┬──────────────────────────┘
                   │ HTTPS + WebSocket
┌──────────────────▼────────────────────────────┐
│              BACKEND (Render)                 │
│        Node.js + Express + Socket.io          │
│  Auth middleware · Matchmaking · Duel state · │
│  Elo engine · Plagiarism detection · AI Coach │
└──────────┬───────────────────┬─────────────────┘
           │                   │ HTTPS + API key
┌──────────▼─────────┐ ┌───────▼──────────────────┐
│     Supabase        │ │   EXECUTION ENGINE       │
│  Postgres + Auth     │ │   (standalone VM)        │
│  RLS-enforced tables  │ │  Docker → Piston, bound  │
│  Users, Matches,     │ │  to localhost, fronted   │
│  Submissions, Reports │ │  by a key-checking proxy │
└────────────────────┘ └──────────────────────────┘
```

The execution engine is intentionally isolated on its own host, with no direct access to the database. Piston has no built-in authentication, so it never faces the public internet directly — it's bound to `localhost` on its VM and reachable only through a small Node proxy that rejects any request without a matching API key. A sandbox escape in the judge should never be able to reach user accounts or ratings.

### Step 1 — Database

In your Supabase project's SQL Editor, run the schema in the [Database Schema](#database-schema) section below, then enable email/password and GitHub providers under Authentication.

---

### Step 2 — Execution Engine

```bash
git clone https://github.com/YOUR-USERNAME/codeduel_execution.git
cd codeduel_execution
```

On your VM: install Docker, run Piston bound to `localhost` only, then run the included proxy in front of it:

```bash
docker run --privileged -v $PWD:/piston -dit -p 127.0.0.1:2000:2000 --name piston_api --restart unless-stopped ghcr.io/engineer-man/piston
export PISTON_PROXY_KEY=$(openssl rand -hex 24)
node index.js
```

Keep that `PISTON_PROXY_KEY` — the backend needs it next.

---

### Step 3 — Backend

```bash
git clone https://github.com/vaibhavdhariwal8/codeduel_backend.git
cd codeduel_backend
npm install
```

Create a `.env` file:

```env
PORT=4000
SUPABASE_JWT_SECRET=your_supabase_jwt_secret
DATABASE_URL=your_supabase_pooler_connection_string
PISTON_PROXY_URL=http://your-vm-ip:3000
PISTON_PROXY_KEY=the_key_from_step_2
GEMINI_API_KEY=your_gemini_api_key
```

```bash
npm run dev
```

You should see:

```
backend on 4000
```

---

### Step 4 — Frontend

Open a new terminal:

```bash
git clone https://github.com/vaibhavdhariwal8/codeduel_frontend.git
cd codeduel_frontend
npm install
```

Create a `.env.local` file:

```env
VITE_SUPABASE_URL=your_supabase_project_url
VITE_SUPABASE_ANON_KEY=your_supabase_anon_key
VITE_BACKEND_URL=http://localhost:4000
```

```bash

```

```sql
create table users (
  id uuid primary key references auth.users(id) on delete cascade,
  username varchar(32) unique not null,
  email varchar(255) unique not null,
  rating int not null default 1200,
  wins int not null default 0,
  losses int not null default 0,
  is_banned boolean not null default false,
  is_admin boolean default false,
  created_at timestamptz not null default now()
);

create table problems (
  id uuid primary key default gen_random_uuid(),
  title varchar(120) not null,
  difficulty varchar(10) not null check (difficulty in ('easy','medium','hard')),
  statement text not null,
  created_at timestamptz not null default now()
);

create table tags (
  id uuid primary key default gen_random_uuid(),
  name varchar(40) unique not null
);

create table problem_tags (
  problem_id uuid references problems(id) on delete cascade,
  tag_id uuid references tags(id) on delete cascade,
  primary key (problem_id, tag_id)
);

create table test_cases (
  id uuid primary key default gen_random_uuid(),
  problem_id uuid references problems(id) on delete cascade,
  visibility varchar(10) not null check (visibility in ('sample','hidden')),
  input text not null,
  expected_output text not null,
  ordinal int not null
);

create table matches (
  id uuid primary key default gen_random_uuid(),
  problem_id uuid references problems(id),
  player_one_id uuid references users(id),
  player_two_id uuid references users(id),
  status varchar(20) not null default 'pending' check (status in ('pending','active','completed','forfeited')),
  result_type varchar(20) check (result_type in ('solved','time_expired','forfeit','draw')),
  winner_id uuid references users(id),
  started_at timestamptz,
  ended_at timestamptz,
  rematch_of_match_id uuid references matches(id),
  created_at timestamptz not null default now(),
  constraint distinct_players check (player_one_id <> player_two_id)
);

create table submissions (
  id uuid primary key default gen_random_uuid(),
  match_id uuid references matches(id),
  user_id uuid references users(id),
  language varchar(20) not null check (language in ('python','javascript','java','cpp')),
  code text not null,
  code_hash varchar(64),
  ai_autopsy jsonb,
  tests_passed int not null default 0,
  tests_total int not null default 0,
  submitted_at timestamptz not null default now()
);

create table submission_test_results (
  id uuid primary key default gen_random_uuid(),
  submission_id uuid references submissions(id) on delete cascade,
  test_case_id uuid references test_cases(id),
  passed boolean not null,
  runtime_ms int,
  ordinal int not null
);

create table rating_history (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references users(id),
  match_id uuid references matches(id),
  rating_before int not null,
  rating_after int not null,
  created_at timestamptz not null default now()
);

create table friendships (
  id uuid primary key default gen_random_uuid(),
  requester_id uuid references users(id),
  addressee_id uuid references users(id),
  status varchar(10) not null default 'pending' check (status in ('pending','accepted','declined')),
  created_at timestamptz not null default now(),
  responded_at timestamptz,
  constraint distinct_friend check (requester_id <> addressee_id)
);

create table custom_duel_invites (
  id uuid primary key default gen_random_uuid(),
  inviter_id uuid references users(id),
  invitee_id uuid references users(id),
  invite_token varchar(64) unique not null,
  difficulty_pref varchar(10) not null default 'any' check (difficulty_pref in ('any','easy','medium','hard')),
  topic_pref varchar(40),
  problem_id uuid references problems(id),
  status varchar(10) not null default 'pending' check (status in ('pending','accepted','expired','cancelled')),
  resulting_match_id uuid references matches(id),
  expires_at timestamptz not null,
  created_at timestamptz not null default now(),
  constraint distinct_invite check (inviter_id <> invitee_id)
);

create table reports (
  id uuid primary key default gen_random_uuid(),
  reporter_id uuid references users(id),
  reported_user_id uuid references users(id),
  match_id uuid references matches(id),
  reason varchar(20) not null check (reason in ('suspected_cheating','harassment','other')),
  details text,
  status varchar(15) not null default 'open' check (status in ('open','reviewed','dismissed')),
  reviewed_by uuid references users(id),
  reviewed_at timestamptz,
  created_at timestamptz not null default now()
);

create table flagged_submission_pairs (
  id uuid primary key default gen_random_uuid(),
  submission_a_id uuid references submissions(id),
  submission_b_id uuid references submissions(id),
  detection_method varchar(20) not null default 'exact_hash',
  status varchar(15) not null default 'open',
  created_at timestamptz not null default now()
);
```

## Key Engineering Highlights

- **Real-time architecture without leaking the thing that matters most** — Socket.IO rooms are scoped per match ID, and the opponent-progress broadcast deliberately carries only pass/fail counts, never code. The entire competitive-fairness premise of the product rests on that one boundary holding under real network conditions, not just in the happy path.
- **Row-Level Security as the actual enforcement layer** — hidden test cases, own-submission-only access, and friend/report visibility rules are enforced by Postgres policies, not just application-layer checks that a future bug could bypass.
- **Defense in depth around untrusted code execution** — the sandbox runs on an isolated host with no direct database access, sits behind an API-key-gated proxy since Piston itself has no authentication, and is bound to `localhost` so it's never directly reachable from the public internet.
- **Plagiarism detection that actually closes the loop** — submissions are hashed and cross-checked against other accounts on the same problem; hits feed a moderation queue that shows admins real signal (distinct reporter count, flagged code matches, head-to-head history) instead of asking them to trust a single free-text claim.
- **Ban enforcement that takes effect immediately, not on next login** — banning a user disconnects their live socket via a personal per-user room and auto-resolves any match they're mid-duel in as a forfeit, rather than leaving a stale "active" match hanging.
- **Elo rating engine** with a standard K-factor, scaling rating deltas by relative opponent strength rather than a flat win/loss point system.

## Notes

- The execution engine is self-hosted and sandboxed, but still runs on free-tier infrastructure — concurrent-duel throughput is intentionally modest, not production-scale
- The backend's free hosting tier spins down after periods of inactivity; the first request after a while can take up to a minute
- Custom Duel currently supports link-based challenges only, not targeted-by-username invites
- Plagiarism detection is exact-hash matching by design — it catches copy-paste, not logic restated with different variable names, which is a deliberately deferred, more involved feature
- No account Settings page yet; account management is limited to logout
